# Fusio Implementation Plan - Overview

## What is Fusio?

Fusio is a Kotlin Compiler Plugin library that decomposes fat Composable Presenters into reusable, self-contained units.

### Problem

In Composable-first architectures (e.g., DroidKaigi 2025), Presenters accumulate `remember` calls, `LaunchedEffect`s, and event handling logic as features grow. This makes them hard to read, test, and maintain.

### Solution

Fusio provides:
- **`buildPresenter`** DSL to define screen-level Presenters
- **`mappedScope`** to delegate to sub-Presenters with automatic Event/Effect type bridging
- **`on<Event>`** for type-safe individual event handling
- **`@MapTo` / `@MapFrom`** annotations for compile-time-safe Event/Effect mapping
- **FIR Checker** for property validation and exhaustiveness checking
- **IR Transformer** for `mappedScope` code generation

### Core API Example

```kotlin
// Screen-level Presenter
context(screenContext: MyScreenContext)
@Composable
fun myScreenPresenter(
    eventFlow: Flow<MyScreenEvent>
): Fusio<MyScreenUiState, MyScreenEffect> = buildPresenter(eventFlow) {
    val favoriteState = mappedScope { favorite() }

    on<MyScreenEvent.Search> { event -> /* handle search */ }

    MyScreenUiState(favorite = favoriteState)
}

// Sub-Presenter (completely independent, reusable)
context(ctx: FavoriteContext)
@Composable
fun PresenterScope<FavoriteEvent, FavoriteEffect>.favorite(): Fusio<FavoriteState, FavoriteEffect> {
    on<FavoriteEvent.Toggle> { event ->
        emitEffect(FavoriteEffect.ShowMessage("Toggled!"))
    }
    return Fusio(FavoriteState(...))
}
```

### Event/Effect Mapping

```kotlin
// Screen Events map to Sub-Presenter Events
sealed interface MyScreenEvent {
    @MapTo(FavoriteEvent.Toggle::class)
    data class ToggleFavorite(val id: String) : MyScreenEvent

    data class Search(val query: String) : MyScreenEvent  // handled locally
}

// Screen Effects map from Sub-Presenter Effects
sealed interface MyScreenEffect {
    @MapFrom(FavoriteEffect.ShowMessage::class)
    data class ShowSnackbar(val message: String) : MyScreenEffect
}
```

## Design Principles

1. **eventFlow is internal** - Users never touch Flow plumbing directly
2. **Sub-Presenters are independent** - No knowledge of parent screen
3. **Context via ISP** - Small interfaces per sub-presenter, composed via inheritance
4. **Context parameters** for dependency propagation - composition is key
5. **Compile-time safety** - @MapTo/@MapFrom validated by FIR, mappedScope generated by IR
6. **Incremental adoption** - Works alongside existing DroidKaigi-style architecture
